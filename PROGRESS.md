- **Connector boundary:** `BaseHttpConnector` holds ONLY HTTP plumbing (rate limit, retry/backoff with jitter on 429/5xx/transport). Each connector owns its envelope parsing, pagination, bounds→params, and writing. The Etherscan adapter (`normalization/etherscan_adapter.py`) is pure (no HTTP/DB) → contract-testable from cassettes. The connector persists via `write_with_provenance` (provenance per call); the orchestrator just dispatches on capability.
- **Provenance per endpoint:** `get_transactions` = 3 source_queries (txlist/txlistinternal/tokentx); `get_balance` = 1. Each stores its raw page payloads (hashed) + bounds in `params`. New **audit #10 (bounds-recorded)** enforces every address-scoped source_query records bounds (or `"default"`).
- **Finality without a tip call:** tip is derived as `max(blockNumber + confirmations)` over txlist+tokentx rows (internal rows have no `confirmations` field — confirmed against live docs), then `finality_for` applied per `algorithms.md §2`. No separate eth_blockNumber call/source_query.
- **Position conventions:** native=0; internal sorted by hierarchical `traceId` then indexed (deterministic across re-fetch → idempotent); erc20 indexed in receipt-log order. - **Multi-endpoint merge:** the same tx is written by up to 3 endpoints; `upsert_transaction` now uses `COALESCE(excluded.x, x)` so txlist's fee/status survive internal's NULLs, while confirmations/finality always refresh (and a `final` row stays frozen). Idempotent re-fetch changes no fact-row counts; claims (balance_snapshot) are append-only (re-fetch = new row).
- **Bounds honesty:** `block_range`/`max_pages`/`direction`/`min_value` are applied; unsupported bounds (`time_window`, `top_n_counterparties`) are **refused loudly** (ConnectorError) rather than silently ignored — deferred to a later phase.

## Follow-ups carried forward

- **Arkham follow-ups (from the 2026-06-28 re-scope):** (1) **cross-source transfer alignment** — Arkham's row-order `position` need not match Etherscan's log-order `position`, so ingesting the same EVM tx from both could create two `transfer` rows for one movement (distinct natural keys, not caught by the idempotency audit). Needs a reconciliation strategy if mixing Arkham+Etherscan on the same txs. (2) **`unitValue` exactness** — it's a UI display value. The `rounded` flag catches only *excess* precision; the likelier loss is Arkham **truncating** a high-decimal token's display (e.g. `1.2346` for `1.234567…`), which yields an integral product and is **not** flagged → the stored `amount` can be silently low-order-lossy. The authoritative amount needs a chain re-fetch, but `upsert_transfer` is `DO NOTHING`, so a later Etherscan fetch won't *correct* an Arkham-first amount (tied to (1)). Treat Arkham EVM amounts as approximate until chain-confirmed. (3) **Arkham `type` / internal calls** — `type` is direction; internal-call representation is unconfirmed. (4) **Arkham BTC as a trace claim** — Arkham's BTC from→to (currently refused) could be ingested as a labeled link inside a `trace` (`basis='arkham'`), never a raw fact. (5) **Path B attribution** — address→entity/label needs the Arkham *API* (Invariant #1), a separate connector. (6) **MisTrack schema** is still the *assumed* CSV columns — confirm against a real MisTrack export (same as the original Arkham CONFIRM-AT-BUILD).
- **Valuation at scale (from the live test):** bulk per-movement DeFiLlama pricing trips the free-tier rate limit on large EVM cases (vitalik.eth → 2,362 transfers). The pass is now robust (skip-not-abort + memo + circuit-breaker + `limit`), but a future improvement is a **batched price layer** — DeFiLlama's endpoint accepts comma-joined keys, and prices could be bucketed by (asset, time-window) and cached in the shared library DB — to value large cases in a handful of calls. Until then, value in bounded batches (`value_movements(..., limit=N)`) or expect honest gaps under rate-limiting.
